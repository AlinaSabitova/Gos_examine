## «Проектирование архитектуры BigData-системы»
### Вариант 2. Финансовая компания

---

## Постановка задачи

Крупная финансовая организация (банк или процессинговый центр) обрабатывает экстремально высокий поток транзакций в реальном времени, а также накапливает исторические данные для оценки рисков. Ключевые бизнес-задачи:
1.  **Выявление мошенничества в реальном времени (Real-time Fraud Detection)** — блокировка подозрительной транзакции до её завершения (в течение < 1 секунды).
2.  **Оценка кредитных рисков (Credit Scoring)** — построение и применение ML-моделей на исторических данных.
3.  **Жесткое регулирование** — соответствие 152-ФЗ и требованиям ЦБ РФ (хранение данных на территории РФ, аудит, защита персональных и финансовых данных).

## Пошаговый алгоритм выполнения

### Шаг 1. Анализ требований

| Параметр | Значение | Архитектурное следствие |
|---|---|---|
| **Объём данных** | **100 ТБ/год**, рост **30%** ежегодно (≈ 371 ТБ через 5 лет) | Требуется массивно-масштабируемое хранилище (HDFS) с учетом геометрического роста |
| **Скорость поступления** | **до 1 000 транзакций/с** | Необходима высокопроизводительная стриминговая шина (Kafka) и потоковый обработчик  (Flink / Spark Streaming) |
| **Типы данных** | 80% структурированные / 15% полу- / 5% неструктурированные | Полиглот-хранилище: быстрый доступ к транзакционным витринам, объектное хранилище для сканов и логов |
| **Обработка** | Реал-тайм маршруты + Оценка рисков | **Kappa-архитектура** (предпочтительно) или Lambda с жестким SLA. Стриминг обязателен. |
| **Доступность** | **99.999%** (~5.26 мин простоя в год) | Гео-резервирование, Multi-AZ кластер, Active-Active или Hot-Standby ЦОД, полное резервирование NameNode и брокеров Kafka |
| **Время отклика** | **< 1 секунды** | In-memory вычисления, K-V хранилище (Redis/Aerospike), процессор правил (Drools) или быстрый ML-инференс |
| **Безопасность** | Сквозное шифрование + **152-ФЗ** + **ЦБ РФ** | Полный аудит (Apache Ranger), шифрование at-rest (HDFS TDE), in-transit (TLS/mTLS), маскирование данных, строгие ролевые модели |

**Расчёт нагрузки**:

```
Поток в реальном времени:
1 000 tx/с × средний размер 4 КБ (JSON с данными карты, MCC, суммой) 
= 4 МБ/с = 345 ГБ/сутки сырых событий
= ~126 ТБ/год только поток (с учетом пиков)

Batch-выгрузки (АБС, архивы):
= ~30 ТБ/год

ИТОГО Сырой объем:
≈ 156 ТБ/год, но с учетом компрессии (snappy/gzip) в Kafka и HDFS — 
эффективно 90-100 ТБ/год ✓ (сходится с условием)

С учетом реплики HDFS = 3:
100 × 3 = 300 ТБ полезной ёмкости дисков в год

Прогноз через 5 лет (рост 30%):
100 × (1.3^5) ≈ 371 ТБ/год → более 1 ПБ дискового пространства с репликацией.
Необходимо масштабирование до 25+ узлов хранения.
```

### Шаг 3. Выбор компонентов архитектуры

#### 3.1. Распределённое хранилище — **HDFS** (основное) + **MinIO/S3** (объектное)

### Шаг 3. Выбор компонентов архитектуры

#### 3.1. Распределённое хранилище — HDFS (основное) + MinIO/S3 (объектное)

**Почему HDFS, а не «голый» S3:**
- Эталон курса (`ds_mgpu_Hadoop3+spark_3_4`) — HDFS, есть локальная экспертиза.
- 152-ФЗ требует размещения ПДн **на территории РФ** → on-premise HDFS закрывает требование без юридических рисков использования зарубежных облаков.
- Поддержка прозрачного шифрования at-rest (HDFS TDE) для выполнения требований ЦБ РФ по защите финансовых данных и банковской тайны.

**Почему дополнительно MinIO (S3-совместимый):**
- Хранение неструктурированных файлов (скан договоров PDF, фото чеков JPG) — для них HDFS неоптимален из-за «small files problem» (перегрузка NameNode множеством мелких файлов).

---

#### 3.2. Шина данных — Apache Kafka + Apache NiFi

- **Kafka** — индустриальный стандарт для приёма 1 000+ транзакций/с, горизонтально масштабируется добавлением брокеров, гарантирует доставку (replication factor = 3, `acks = all`, `min.insync.replicas = 2`). Долговременное хранение лога событий (retention) необходимо для аудита операций по требованиям ЦБ РФ.
- **NiFi** — графический конструктор для сбора из разнородных источников (ISO 8583 от процессинга, HTTP/REST от мобильного банка, JDBC от АБС, файловые загрузки). Выполняет токенизацию номеров карт и маскирование ПДн «на лету», до попадания в аналитический контур.

---

#### 3.3. Потоковая обработка реального времени — Apache Flink

- **Flink** выбран для контура антифрода. Обеспечивает событийно-ориентированную (event-by-event) обработку без накопления микробатчей — задержка менее 200 мс, что укладывается в SLA «< 1 с» с пятикратным запасом.
- Нативная библиотека **CEP (Complex Event Processing)** позволяет выявлять сложные паттерны мошенничества на временных окнах: «5 транзакций по одной карте в разных городах за 1 минуту», «сумма покупок за час превысила среднемесячный профиль клиента в 10 раз».
- Сохранение состояния в **Savepoints** на HDFS обеспечивает восстановление обработки ровно с места сбоя без потери событий — требование доступности 99.999%.

---

#### 3.4. Пакетная обработка и ML — Apache Spark (PySpark) + MLflow

- **Spark** выбран для batch-контура: переобучение моделей кредитного скоринга на исторических данных (3 года транзакций), расчёт витрин для отчётности ЦБ, Feature Engineering.
- **Spark MLlib** — обучение моделей (Gradient Boosting, Logistic Regression) для оценки вероятности дефолта заёмщика.
- **MLflow** — версионирование моделей и трекинг экспериментов. Обученная модель экспортируется в формат PMML и загружается в контур Flink для использования в потоковых правилах антифрода.

---

#### 3.5. Витрины данных — Hive/Iceberg + ClickHouse

- **Hive / Iceberg** — SQL-доступ поверх HDFS для аналитиков и Data Scientists. Iceberg обеспечивает ACID-транзакции и Snapshot Isolation — критично для регламентной отчётности ЦБ (форма 0409135 и др.) на фиксированный срез данных без влияния продолжающейся потоковой записи.
- **ClickHouse** — субсекундные ответы для оперативных дашбордов мониторинга фрода: количество срабатываний правил, гео-карта подозрительной активности, топ торговых точек по отклонённым транзакциям.

---

#### 3.6. In-memory кэш — Redis

- **Redis** — горячий кэш для контура антифрода. Хранит чёрные списки украденных карт, текущие лимиты по счетам, гео-профили клиентов. Время ответа < 1 мс необходимо для удержания сквозной задержки всей проверки в пределах 1 секунды.

---

#### 3.7. Оркестрация — Apache Airflow

- DAG-и расписывают ночные ETL: загрузка выгрузок из АБС (Core Banking), переобучение моделей кредитного скоринга, построение витрин в Iceberg, формирование регламентных отчётов для ЦБ РФ, архивирование аудит-логов старше 5 лет на холодное хранение.

---

#### 3.8. Безопасность (под 152-ФЗ и требования ЦБ РФ)

| Компонент | Назначение |
|---|---|
| **Kerberos** | Аутентификация всех сервисов кластера (Hadoop, Kafka, Hive, Flink). Исключает несанкционированное подключение узлов |
| **Apache Ranger** | RBAC + ABAC: разграничение доступа к таблицам и колонкам. Тэги `PAN` (номер карты), `PII` (персональные данные). Data Scientist видит только токен вместо реального номера карты |
| **mTLS** | Взаимная аутентификация и шифрование in-transit между всеми компонентами (Kafka ↔ Flink, NiFi ↔ Kafka). Каждый узел предъявляет сертификат |
| **HDFS TDE** | Прозрачное шифрование at-rest для зон хранения ПДн и финансовых данных. Ключи управляются через KMS |
| **Токенизация в NiFi** | Замена реальных номеров карт на обезличенные токены на этапе приёма, до записи в HDFS |
| **Аудит-логи Ranger** | Журналирование всех фактов доступа к конфиденциальным данным — требование 152-ФЗ и СТО БР ИББС |
| **Размещение в РФ** | Основной и резервный ЦОД на территории России. Срок хранения аудит-логов — не менее 5 лет |
